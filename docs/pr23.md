# PR #23: Azure Entra ID / Managed Identity authentication

## What this PR does

Adds a second Azure Blob Storage authentication backend that uses Azure's
token-based credential chain (Entra ID) instead of a connection string. This
lets sccache authenticate via managed identity on Azure VMs/AKS/App Service,
via service-principal environment variables, or via Azure CLI â€” without
embedding storage account keys in config.

The existing connection-string path (`SCCACHE_AZURE_CONNECTION_STRING`) is
unchanged. The new path is activated by setting `SCCACHE_AZURE_BLOB_ENDPOINT`
instead. Both require `SCCACHE_AZURE_BLOB_CONTAINER`.

A CI fix is also included: the pinned Rust 1.82.0 toolchain used to install
`grcov` broke because grcov's dependencies now require Rust 1.88+. The fix
removes the pin so it uses the stable toolchain already present in CI.

## Data model changes

```rust
// Before
pub struct AzureCacheConfig {
    pub connection_string: String,       // required
    pub container: String,
    pub key_prefix: String,
}

// After
pub struct AzureCacheConfig {
    pub connection_string: Option<String>,          // now optional
    pub container: String,
    pub key_prefix: String,
    pub storage_account_endpoint: Option<String>,   // new: blob endpoint URL
}
```

Exactly one of `connection_string` or `storage_account_endpoint` must be set.
If neither is present, Azure cache is disabled. If both are set,
`connection_string` takes priority (checked first in `storage_from_config`).

**Why two separate backends instead of extending AzureBlobCache?** The existing
`AzureBlobCache` uses OpenDAL's `Azblob` operator which is wired to
connection-string auth. The credential-chain path needs the Azure SDK's
`BlobClient` which accepts a `TokenCredential`. Rather than trying to shoehorn
a credential chain into OpenDAL's abstraction, the new backend implements the
`Storage` trait directly against `azure_storage_blob`. This keeps the existing
path untouched and avoids coupling the two auth mechanisms.

**Why a custom `ChainedCredential` instead of Azure SDK's
`DefaultAzureCredential`?** The Azure SDK for Rust doesn't ship a
`DefaultAzureCredential` type. The `ChainedCredential` here replicates the
same priority order that Azure SDKs in other languages use.

**How `check()` works:** The `Storage::check()` method probes access by
calling `get_properties` on a sentinel blob (`.sccache_check`). A 404 is
treated as "readable" (the container exists and we have read access; the blob
just isn't there). It then attempts a small upload to test write access. If the
upload fails (e.g. read-only RBAC role), the cache degrades to `ReadOnly`
mode rather than failing entirely.

## New modules/functions

### `src/cache/azure.rs`

| Item | Purpose |
|---|---|
| `ChainedCredential` | Tries multiple Azure credential sources in priority order |
| `ChainedCredential::new()` | Builds the chain from env vars and available identity sources |
| `ChainedCredential::get_token()` | Iterates sources until one succeeds |
| `AzureBlobCredentialCache` | `Storage` impl using Azure SDK + credential chain |
| `AzureBlobCredentialCache::build()` | Constructor: creates credential chain, stores endpoint/container/prefix |
| `AzureBlobCredentialCache::blob_client()` | Creates a `BlobClient` for a specific blob path |
| `AzureBlobCredentialCache::blob_path()` | Maps cache key to blob path (prepends key_prefix) |
| `Storage::get` impl | Downloads blob, returns `Cache::Hit` or `Cache::Miss` |
| `Storage::put` impl | Uploads serialized cache entry to blob |
| `Storage::check` impl | Probes read/write access via sentinel blob |

## Security considerations

- Credential-chain env var values (`AZURE_CLIENT_SECRET`) are never logged; debug output shows `<set>` / `<not set>` only.
- `connection_string` values are also redacted in debug logs.
- The `check()` sentinel blob upload uses a fixed harmless payload (`"Hello, World!"`), not cache data.
